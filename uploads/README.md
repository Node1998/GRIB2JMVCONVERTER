# JMV Auto-Ingest Service

Pulls the latest ECMWF open-data model cycle every 6 hours, converts GRIB2 →
JMV, and serves the output over HTTP. Extracted from the `GRIB2JMV_GUIDE`
notebook's streamlined converter.

## Layout

| File            | Role |
|-----------------|------|
| `converter.py`  | GRIB2→JMV core (ParamSpecs, header, writer, bundle converter). No Flask. |
| `config.py`     | All env-driven config + model→backend map. |
| `ingest.py`     | Cycle pull + convert + prune. Single-flight lock, CLI entry point. |
| `app.py`        | Flask routes: `/health`, `/cron/ingest`, `/files`. |
| `Dockerfile`    | Python 3.11 + native eccodes. |
| `render.yaml`   | Web service (owns disk) + cron trigger (self-curl). |

## How the schedule works

Render cron jobs and web services can't share a persistent disk. So the web
service owns the disk and does the work; the cron container just POSTs
`/cron/ingest` every 6h. The route returns `202` immediately and runs the pull
on a background thread (Render's request timeout is ~30s; a full pull is minutes).

Schedule is `5 */6 * * *` (`:05` past every 6th hour) because ECMWF open data
lags the cycle by ~7–9h — `ingest.py` probes `client.latest()` and pulls
whatever run is actually posted, not the one that just started.

## Endpoints

| Method | Path                        | Purpose |
|--------|-----------------------------|---------|
| GET    | `/health`                   | Liveness + last-run status. |
| POST   | `/cron/ingest`              | Trigger a pull. Requires `X-Cron-Secret`. |
| GET    | `/files`                    | List cycles → JMV filenames. |
| GET    | `/files/<cycle>/<name>`     | Download one JMV. |

## Local run

```bash
cp .env.example .env          # set JMV_DATA_DIR to a local path
pip install -r requirements.txt
export $(grep -v '^#' .env | xargs)
python -m ingest              # one synchronous pull (needs eccodes installed)
# or serve:
gunicorn -w 1 -t 600 -b 0.0.0.0:8000 app:app
```

## Deploy (Render)

1. Push to a Git repo.
2. New → Blueprint → point at the repo (`render.yaml` auto-detected).
3. `CRON_SECRET` is generated by Render and shared to the cron service.
4. Edit the web-service URL in `render.yaml`'s `dockerCommand` to your actual
   `*.onrender.com` host before first deploy.

## Config knobs (env vars)

| Var             | Default        | Notes |
|-----------------|----------------|-------|
| `ECMWF_MODEL`   | `HRES`         | `HRES` \| `AIFS` \| `ENSEMBLE` \| `WAVE`. |
| `ECMWF_RES`     | `0.25/0.25`    | AIFS is 0.25° only. |
| `MAX_FCST_HR`   | `72`           | Longest forecast hour to pull. |
| `STEP_INTERVAL` | `6`            | Hours between forecast steps. |
| `RETENTION_HRS` | `48`           | Cycles older than this are pruned. |
| `CRON_SECRET`   | *(empty)*      | **Set in prod** — gates the ingest route. |

## Scaling note

`-w 1` (single gunicorn worker) is deliberate: the ingest lock and last-run
status live in process memory. To run multiple workers, move that state to the
persistent disk or a DB (e.g. a small SQLite manifest of ingested cycles) so
the single-flight guard and idempotency check stay correct across workers.

## Security / integrity

- Ingest route is secret-gated (constant-time compare) — otherwise the
  expensive pull is world-triggerable (cost / rate-limit DoS).
- Single-flight lock returns `409` if a pull overlaps the next cron tick.
- Idempotent per cycle: re-triggering an already-ingested cycle is a no-op, so
  cron retries are safe.
- File serving uses `send_from_directory` (blocks `../` traversal); cycle
  segment validated as an existing dir.

## Not included (optional extensions)

- SQLite manifest for fast `/files` + cross-worker state.
- `?force=1` to re-ingest a cycle.
- Multi-region grid subsetting (`area=[N,W,S,E]` on `retrieve`).
- Notification hook (Slack/email) on ingest failure.
